# Рецензия на реализацию AGMY-RAG Admin Panel

## Общая оценка

Проект реализован как полноценный full-stack веб-сервис администрирования для RAG-системы. Структура чистая, разделение ответственности соблюдено, все заявленные в плане модули присутствуют и работают. Клиент успешно собирается Vite-ом в production-бандл без ошибок, зависимости установлены без конфликтов.

---

## Бэкенд (Node.js + Express)

### Что сделано хорошо

**Аутентификация** реализована по всем стандартам безопасности:
- Access-токен хранится только в памяти клиента (не в `localStorage`, не в `sessionStorage`)
- Refresh-токен передаётся исключительно через `httpOnly`-куку, недоступную для JavaScript
- Применяется ротация токенов: при каждом обновлении старый refresh-токен удаляется из БД и выдаётся новый
- Хеши refresh-токенов хранятся в PostgreSQL через `SHA-256`, а не сами токены — компрометация базы не позволит использовать токены напрямую

**Маршруты документов** корректно разделяют синхронный и асинхронный слои: ответ `202 Accepted` возвращается клиенту немедленно, а индексация через Python запускается в фоне. Статус документа (`processing → indexed / error`) обновляется по результату работы subprocess.

**Маршруты вопросов** поддерживают полноценную пагинацию с динамической генерацией `WHERE`-условий, что позволяет комбинировать фильтры (`search`, `category`, `is_active`) без SQL-инъекций — параметры передаются через `$N`-placeholders библиотеки `pg`.

**Python-сервис** (`services/python.js`) использует `child_process.spawn` (не `exec`), что исключает shell-инъекции через имена файлов. Пути настраиваются через переменные окружения.

### Спорные моменты и ограничения

**Multer сохраняет файлы напрямую в `knowledge_base/` Python-проекта** — это работает, если оба сервиса запущены на одной машине. В облачном или контейнеризированном окружении потребуется общий volume или объектное хранилище (S3-совместимое). В `docker-compose.yml` это учтено через `${AI_SUPPORT_PATH}`, но требует ручной настройки.

**Инициализация `KB_PATH` в `upload.js`** происходит при каждом запросе (`getUploadDir()`), а не один раз при запуске сервера. Это нормально для development, но в production лучше вынести в инициализацию middleware.

**Нет rate limiting** на `/api/auth/login` — при желании можно добавить `express-rate-limit`, чтобы защитить эндпоинт от brute-force атак.

**Нет валидации размера файла на уровне Express** до того, как multer начнёт запись на диск. Лимит в 50 МБ задан в опциях multer и сработает корректно, но сервер будет принимать поток данных до момента превышения.

## База данных (PostgreSQL)

### Что сделано хорошо

- Схема нормализована: 4 отдельные таблицы с чёткой ответственностью
- `updated_at` на таблице `questions` обновляется автоматически через `TRIGGER` — не нужно помнить об этом на уровне приложения
- Индексы созданы на `category`, `is_active` (вопросы) и `token_hash`, `user_id` (refresh-токены) — запросы по этим полям будут быстрыми даже при больших объёмах
- Seed-скрипт использует `ON CONFLICT DO UPDATE`, что делает его идемпотентным — повторный запуск не сломает данные

### Ограничения

**Нет версионирования миграций** — только один файл `001_init.sql`. При расширении схемы в будущем придётся либо вести нумерованные файлы вручную, либо подключать инструмент типа `node-pg-migrate` или `Flyway`.

**`refresh_tokens` не имеет индекса по `expires_at`** — при большом числе записей периодическая чистка устаревших токенов будет медленной. Рекомендуется добавить индекс и cron-задачу очистки.

## Фронтенд (React + Tailwind CSS)

### Что сделано хорошо

**Архитектура состояния** выбрана оптимально:
- `Zustand` — для глобального auth-состояния (лёгкий, без boilerplate)
- `TanStack Query` — для серверного состояния (кэш, фоновое обновление, `keepPreviousData` для плавной пагинации)
- Локальный `useState` — для UI-состояний (модалки, формы)

**Автоматическое обновление токена** (`api/client.js`) реализовано корректно: конкурентные запросы, прилетевшие во время обновления токена, ставятся в очередь и выполняются после получения нового access-токена — не теряются и не дублируют refresh-запрос.

**`KnowledgeBasePage`** автоматически опрашивает сервер каждые 5 секунд (`refetchInterval`), пока хотя бы один документ имеет статус `processing` — это даёт пользователю живую обратную связь о прогрессе индексации без ручного обновления страницы.

**Drag-and-drop зона** корректно обрабатывает события `dragover`, `dragleave`, `drop` и даёт визуальную обратную связь через изменение стилей. Прогресс загрузки отображается через `onUploadProgress` axios.

**Компоненты `Modal` и `ConfirmDialog`** правильно закрываются по клику на backdrop и нажатию `Escape` — стандартное UX-поведение.

### Спорные моменты и ограничения

**`QuestionsPage` использует устаревший API `keepPreviousData`** — в TanStack Query v5 это свойство было переименовано в `placeholderData: keepPreviousData`. Функционально работает через обратную совместимость, но генерирует предупреждение. Рекомендуется заменить на:
```js
import { keepPreviousData } from '@tanstack/react-query'
// ...
placeholderData: keepPreviousData,
```

**Нет сохранения фильтров QuestionsPage** в URL (`?search=...&page=2`) — при обновлении страницы фильтры сбрасываются. Для production-уровня рекомендуется использовать `useSearchParams` из React Router.

**Форма вопроса не сбрасывается автоматически** при закрытии модалки кнопкой backdrop/Escape — состояние `form` остаётся в последнем значении до следующего открытия (компенсируется тем, что `openCreate` явно сбрасывает форму в `EMPTY_FORM`, а `openEdit` заполняет её данными строки).

**Нет обработки ошибки сети** на уровне глобального UI — если сервер недоступен, пользователь видит только `react-hot-toast` с сообщением об ошибке, без явного индикатора "сервер недоступен".

## Интеграция с Python RAG

### Что сделано хорошо

**Инкрементальная индексация** — основная задача выполнена правильно. Вызов `ingest.py` на уже проиндексированных файлах не тратит токены, так как SHA-256 дедупликация в `ingest.py` пропустит уже существующие чанки.

**`admin_tools.py`** написан оборонительно: обрабатывает отсутствие коллекции, пустой результат запроса, принимает `--db-path` как параметр (не хардкодит путь).

### Ограничения

**ChromaDB `$contains` на поле `source`** может работать медленно при большом числе чанков — ChromaDB не поддерживает стандартные SQL-индексы на метаданных. Альтернатива: хранить `source` как точное значение и использовать `$eq` вместо `$contains`, если пути всегда полные.

**Нет обработки случая, когда Python-окружение недоступно** (не установлены зависимости `chromadb`, нет `python3` в PATH) — сервер вернёт ошибку, которая будет записана в `error_message` документа. Это приемлемо, но стоит добавить health-check эндпоинт (`GET /api/health`) для проверки доступности Python при старте.

## DevOps и развёртывание

### Что сделано хорошо

- `docker-compose.yml` корректно задаёт зависимость `server` от `postgres` через `healthcheck`, а не просто `depends_on` — сервер не стартует до готовности базы
- Production-сборка клиента выводится напрямую в `server/public/`, что позволяет Express отдавать SPA без отдельного nginx
- `Dockerfile` для сервера использует `npm ci --omit=dev` — минимальный образ без dev-зависимостей

### Ограничения

**Нет `.dockerignore`** — при сборке образа в контекст попадают `node_modules/`, что замедляет build и увеличивает контекст.

**`docker-compose.yml` использует `version: '3.9'`** — это поле устарело в последних версиях Docker Compose v2 и генерирует предупреждение. Рекомендуется убрать поле `version` полностью.

**Нет production-образа для клиента** — `Dockerfile.dev` запускает Vite dev-сервер, который не предназначен для production. В production клиент собирается через `npm run build` и раздаётся Express-сервером, что правильно, но явный production Dockerfile для клиента мог бы быть полезен.

## Безопасность: итоговая оценка

| Аспект | Статус | Комментарий |
|---|---|---|
| Хранение токенов | Хорошо | Access — в памяти, Refresh — httpOnly cookie |
| Ротация токенов | Хорошо | Реализована полностью |
| SQL-инъекции | Хорошо | Параметризованные запросы везде |
| Command injection | Хорошо | `spawn` вместо `exec`, пути из env |
| CORS | Хорошо | Настроен на конкретный origin |
| Brute-force защита | Отсутствует | Нет rate limiting на `/auth/login` |
| HTTPS | Не настроен | Ожидается настройка на уровне reverse proxy |
| Валидация загружаемых файлов | Частичная | Проверяется MIME и расширение, но не содержимое |
